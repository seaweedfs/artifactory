# S3 Gateway

Testing the SeaweedFS **S3 gateway** — bucket/object round-trips against the S3
REST API.

**Binary:** `sweeds3` (core + `s3` pack), or the all-in-one `sw-test-runner`.

## Quick start

```bash
make build-sweeds3           # or: go build ./cmd/sweeds3

sweeds3 run scenarios/s3-smoke-chain.yaml \
  -results-dir /mnt/smb/work/share/testops/results/s3-qa
```

## The reference scenario

`scenarios/s3-smoke-chain.yaml` is a full `build → seed → test → cleanup` chain,
proven on m02 (10/10 actions, zero residue):

1. **build** — `git clone` SeaweedFS + `go build weed` (the repo has replace
   directives, so `go install …@latest` fails — clone + build is required).
2. **seed** — `s3_start_stack` (`weed server -s3 -filer …`, waits for HTTP) +
   `s3_make_bucket`.
3. **test** — `s3_verify_roundtrip` (put random object, get, assert md5) +
   `s3_list` + `assert_greater`.
4. **cleanup** (`always: true`) — `s3_stop_stack` (waits for the process to exit)
   + a zero-residue process check.

## Pack actions (`packs/s3`)

`s3_start_stack`, `s3_stop_stack`, `s3_make_bucket`, `s3_put_object`,
`s3_get_object`, `s3_list`, `s3_verify_roundtrip` — all curl-against-REST, **no
aws-cli dependency**. The weed S3 gateway allows anonymous PUT/GET over plain
curl for smokes; production suites add SigV4 creds / ceph-s3-tests / warp via
`exec`.
